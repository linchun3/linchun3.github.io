typing with python

“If it walks like a duck, and it quacks like a duck, then it must be a duck.”

Dynamic typing(Duck typing in particular) is one of the main reasons why python is so popular today. The learning curve for python much lower than static-typed languages like C and Java, because there is less “boiler-plate” involved in writing code.

Having experienced both static and dynamic typed languages, I have come to appreciate the assurances provided by static typed languages, which has improved my development experience.

Although Python is fundamentally a dynamic typed language, it has introduced some form of typing in Python 3.5. Although it is not fully enforced, there are 3rd party tools that help to check for type errors.

def greeting(name: str) -> str:
    return 'Hello ' + name

Static analysis tools like mypy help to parse code and determine errors, even before the code is executed in runtime.

greeting(3)         # Argument 1 to "greeting" has incompatible type "int"; expected "str"
greeting(b'Alice')  # Argument 1 to "greeting" has incompatible type "bytes"; expected "str"

I usually have these 3rd party tools installed as a pre-commit hooks, which runs through the hooks prior to a commit.

The introduction of type hinting in python has helped me to write code that is less ambigious and error-prone. Just as with the introduction of typescript, typing hinting is also not compulsory. This means that it’s still as easy to write python code as a beginner and then progressively introduce type hinting.

Next up, I’ll probably be exploring Protocol and Abstract Base Classes (ABC) to create interfaces and contracts.
