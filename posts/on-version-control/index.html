on version control

I used to fear the use of git, especially on projects that involved multiple stakeholders with no protection.

Fast forward to today, I version control all of my projects (dotfiles, blogs, school work, tutorials etc…) and I’m no longer afraid to work through merge conflicts and rebases.

It all started with me trying to understand more about git, through Git Pro. It is a book filled with insights and probably the most underrated book I’ve ever read thus far.

I believe that understanding how git works is fundamental for all engineers.

Here are the chapters that I thought interesting:

  1. What is git
  2. Undoing things
  3. Branching workflows
  4. Rebasing
  5. Stashing
  6. Git Hooks

For most projects, I follow GitHub flow, which I find simpler and easier to work with.

My usual process goes like this:

  1. Fetch and merge in new changes (git pull --rebase)
  2. Create new branch (git checkout -b branchname)
  3. Work on new branch
  4. Stage changes in hunks
  5. Write good commits
  6. Push to remote
  7. Create PR to merge to main.
