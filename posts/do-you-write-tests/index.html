do you write tests?

My conversations among peers in the engineering field rarely touch upon testing.

It’s not all that surprising because who would want to talk about a boring topic or a practice that is does not exist at their workplace?

I like to broach the topic of testing because it gives me a glimpse into how different engineering teams perceive testingand how much they value their engineering craft.

Some engineers test their code, even if they do not explicitly write tests. They often find writing tests tedious and that they do not provide additional value to their feature.

I concede that writing tests can be tedious, but it is definitely not useless.

Without writing tests, how would you know that newly introduced change does not break your existing code? Without writing tests, how would you be sure that you’ve tested a wide range of edge cases?

Some engineers also argue that testing, in particular [unit testing is useless](Unit Testing is Overrated | Alexey Golub (https://tyrrrz.me/blog/unit-testing-is-overrated)) and contributes zero value to development.

I strongly disagree.

My opinion is that if you find unit testing useless, then you’re probably not doing it right. There could be many reasons, one of which is that unit testing can be difficult as a purist insists that each dependency has to be abstracted away. The definition of unit testing can be subjective because the term unit itself is subjective. I treat a unit as a logical segment of code that does one thing, in this case if I perform functional programming, then a unit is a function.

In my career, I’ve been to places where written tests were non-existent, whenever I had to push code, there would always be this looming fear of your code breaking in production and having to debug and fix it on the fly.

At my current workplace, we have automated testing in place and a culture to write(and fix) tests for every piece of code written. It has helped me to feel more confident of introducing changes and through the process of fixing code to pass tests, it has helped me to write better code.

I was greatly influenced by this TDD book on writing tests. What I enjoyed most was the walkthrough of “red, green, refactor” with a singular example throughout the book.

Maybe, a few years down the road I might form a different opinion on testing. Who knows?
